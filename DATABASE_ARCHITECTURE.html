🔗 Core Architecture

All mission apps share a single PostgreSQL database. The Site model is the central multi-tenant entity. Every user and resource is scoped to one or more sites for data isolation.

CORE Site

Multi-tenant root entity. Every user and resource belongs to a site.

id String (PK)
label String
createdAt, updatedAt DateTime

CORE User

Hub-authenticated user. Linked to primary site and array of allowed sites.

id String (PK)
username, email String (UNIQUE)
passwordHash String
siteId, siteIds[] Site relationship
hubRole, jobRole String
status, role, tokenEpoch String, Int

🏢 Multi-Tenant Architecture

Site Scoping Pattern

Every resource record includes a siteId foreign key that references the Site table.

Users can be assigned to multiple sites via the User.siteIds[] array, enabling:

  • Corporate admins to see all sites (via canSeeAllSites: true)
  • Site-scoped users to see only their assigned site(s) (filtered by requiredAuth middleware)
  • All data access to be filtered at the database query level, not in application logic

Authorization Model

  • Hub Role: Hub Admin (full access), Hub Viewer (read-only), etc.
  • Job Role: App-specific role (e.g., "RMF Officer", "Security Manager")
  • Site Assignment: Primary site + array of multi-tenant sites
  • App Access: Array of allowed apps per user (SCORVA, CRATER, MASH, LAVA, etc.)

🔄 Data Flow & Relationships

SCORVA Flow

User selects a Site → Views Controls for that Site → Creates Tasks linked to Controls → POAMs track remediation → ConMon tracks compliance

CRATER Flow

Create ProjectSystem (e.g., "Email Server") → Add ProjectControls from StandardLibrary → Gather evidence via EvidenceArtifacts → Validate controls → Generate SSP

MASH Flow

Register Site as MashSite → Manage Facility/Personnel/Media security → Track DD-254 contracts → Document clearances → Log findings → Measure compliance

LAVA Flow

Submit SAAR request (LavaSaar) → Assign hardware assets (LavaAssets) → Track assignment history → Audit all changes (LavaAuditLog)

📋 Summary

Total Tables: 50+

Primary Key Strategy: CUID/UUID + String IDs for interoperability

Multi-Tenancy: Site-scoped via siteId foreign key on all mission resources

Authentication: Express-session (Hub) + JWT (other apps) with httpOnly cookies

Authorization: Role-based (hubRole) + Site-based (siteIds[]) + App-based (allowedApps[])

Domains: SCORVA (RMF), CRATER (eMASS), MASH (Security), LAVA (Network), NEXUS (Analytics)

Tech Stack: PostgreSQL 16 + Prisma ORM + Node.js
