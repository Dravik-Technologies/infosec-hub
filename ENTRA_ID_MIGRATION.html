🔐 Entra ID 100% Migration Guide

Azure Active Directory Single Sign-On for All Mission Apps

Quick Answer: You do NOT need separate databases per site. Single PostgreSQL with Entra groups for site assignment is simpler and more secure. Multi-tenancy is enforced at the application layer, not the database layer.

🏗️ Architecture Overview

Current Architecture (Local Auth + Sessions)

User → Hub Login Form (admin/admin) ↓ express-session cookie (hub.sid) ↓ req.session.user populated from DB ↓ Each app validates session or JWT ↓ Site scoping via siteId in WHERE clause

New Architecture (100% Entra ID)

User → Azure Entra ID Login ↓ MSAL gets JWT from Entra ↓ JWT stored in localStorage (MSAL cache) ↓ All requests: Authorization: Bearer {JWT} ↓ Backend: Verify JWT signature, extract claims ↓ Claims include Entra groups (site assignment) ↓ Each app enforces site scoping from claims

Database Topology

❌ Multi-DB per Site (DON'T DO THIS)

  • Site A → site_a_pg
  • Site B → site_b_pg
  • Site C → site_c_pg
  • Problems:
  • Migration complexity (copy schemas N times)
  • No cross-site rollups (NEXUS analytics)
  • Duplicate master data (users, configs)
  • Operational overhead

✅ Single DB + Entra Groups (DO THIS)

  • One PostgreSQL for all sites
  • Entra groups control site access
  • siteId filtering at query layer
  • Benefits:
  • Simpler operations & backups
  • Cross-site analytics work
  • Single source of truth for shared data
  • Less infrastructure cost

2. Entra Token Verification Module

// lib/entraAuth.js import jwt from 'jsonwebtoken'; import JwksClient from 'jwks-rsa'; const jwksClient = JwksClient({ jwksUri: `https://login.microsoftonline.com/${process.env.ENTRA_TENANT_ID}/discovery/v2.0/keys`, }); async function getSigningKey(kid) { const key = await jwksClient.getSigningKey(kid); return key.getPublicKey(); } export async function verifyEntraToken(token) { try { const decoded = jwt.decode(token, { complete: true }); if (!decoded) throw new Error('Invalid token format'); const kid = decoded.header.kid; const signingKey = await getSigningKey(kid); const verified = jwt.verify(token, signingKey, { issuer: `https://login.microsoftonline.com/${process.env.ENTRA_TENANT_ID}/v2.0`, audience: process.env.ENTRA_APP_ID, algorithms: ['RS256'], }); return verified; } catch (err) { throw new Error(`Token verification failed: ${err.message}`); } } // Extract site IDs from Entra groups export function extractSitesFromClaims(claims) { // Entra groups come as array in claims.groups // Format: "SITE_A_USERS", "SITE_B_USERS", etc. const groups = claims.groups || []; const siteIds = groups .filter(g => g.startsWith('SITE_')) .map(g => g.replace('SITE_', '').replace('_USERS', '')) .map(s => s.toLowerCase()); return siteIds; }

3. New requireAuth Middleware

// middleware/requireAuth.js import { verifyEntraToken, extractSitesFromClaims } from '../lib/entraAuth'; export async function requireAuth(req, res, next) { try { const authHeader = req.headers.authorization || ''; if (!authHeader.startsWith('Bearer ')) { return res.status(401).json({ error: 'Missing or invalid Authorization header' }); } const token = authHeader.slice(7); const claims = await verifyEntraToken(token); // Build user object from Entra claims req.user = { id: claims.oid, // Entra object ID email: claims.email || claims.preferred_username, name: claims.name, username: claims.preferred_username, hubRole: 'Hub Admin', // Derive from Entra groups/roles siteIds: extractSitesFromClaims(claims), primarySiteId: extractSitesFromClaims(claims)[0] || null, // Pass token for downstream validation token: token, tokenExpiry: claims.exp, }; next(); } catch (err) { console.error('[Auth Error]', err.message); return res.status(401).json({ error: 'Unauthorized' }); } }

🔒 How Site Data Isolation Works

Flow Diagram

1. User logs in with Entra ID
↓
2. Entra returns JWT with groups claim
groups: ["SITE_A_USERS", "SITE_B_USERS"]
↓
3. App extracts siteIds from groups
siteIds: ["a", "b"]
↓
4. Every database query adds WHERE clause
WHERE siteId IN ('a', 'b')
↓
Result: User only sees their site data

Example: Admin Console Access Control

// hub/server/routes/admin.js app.get('/api/admin/users', requireAuth, async (req, res) => { // Only HubAdmin can access if (!req.user.roles?.includes('HubAdmin')) { return res.status(403).json({ error: 'Admin access required' }); } // HubAdmin sees all users const users = await db.user.findMany(); res.json(users); }); app.get('/api/admin/users/by-site', requireAuth, async (req, res) => { // Site admins can only view users in their sites const users = await db.user.findMany({ where: { siteIds: { hasSome: req.user.siteIds, // User sees only overlapping sites }, }, }); res.json(users); });

Example: SCORVA Control Access

// scorva/server/routes/controls.js app.get('/api/controls/:id', requireAuth, async (req, res) => { const control = await db.control.findUnique({ where: { id: req.params.id }, }); // Enforce site isolation if (!req.user.siteIds.includes(control.siteId)) { return res.status(403).json({ error: 'Access denied' }); } res.json(control); });

⚠️ Common Pitfalls & Solutions

1. Token Expiry Handling

Pitfall: User's JWT expires, app doesn't refresh, user gets 401
// Solution: Use MSAL's acquireTokenSilent with error handling axiosInstance.interceptors.response.use( response => response, async error => { if (error.response?.status === 401) { try { const token = await instance.acquireTokenPopup({ scopes: ['api://app-id/access'], }); // Retry request with new token error.config.headers.Authorization = `Bearer ${token.accessToken}`; return axios(error.config); } catch (refreshError) { // Force re-login instance.logout(); } } return Promise.reject(error); } );

2. Group Synchronization Delay

Pitfall: User added to Entra group, but token still shows old groups (Entra cache delay)

Solution: Entra caches group claims for ~30 minutes. Either:

  • Tell users to log out and back in
  • Implement manual token refresh UI
  • Reduce token cache time (less secure, more calls)

3. Cross-Site Admin Operations

Pitfall: Hub Admin creates user, forgets to add to site group, user sees no sites

Solution: In admin UI, show group membership and allow adding users to groups directly from app.

4. NEXUS Cross-Site Rollups

Pitfall: NEXUS should sum metrics across all sites, but now filtered by user's siteIds
// Solution: NEXUS queries check user role app.get('/api/nexus/rollup', requireAuth, async (req, res) => { let whereClause = {}; // If not HubAdmin, filter to user's sites only if (!req.user.roles?.includes('HubAdmin')) { whereClause.siteId = { in: req.user.siteIds }; } // If HubAdmin, leave empty (gets all sites) const snapshot = await db.nexusSnapshot.aggregate({ where: whereClause, _sum: { openPoams: true, authorizedAtos: true }, }); res.json(snapshot); });
